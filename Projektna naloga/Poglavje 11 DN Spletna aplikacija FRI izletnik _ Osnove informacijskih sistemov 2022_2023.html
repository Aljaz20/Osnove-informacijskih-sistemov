11.1 Spletna aplikacija (1. del)

11.1.1 Opis naloge in navodila

V GitHub repozitoriju DN-{študent}, v okviru organizacije našega predmeta ois-2022-2023, kjer {študent} predstavlja GitHub uporabniško ime posameznega študenta, je na voljo podobna Node.js spletna prodajalna, kot jo bomo spoznali na vajah V6 Spletna trgovina e-Pesmi, za razliko, da se spletna trgovina nanaša na domeno turističnih destinacij, in sicer turističnih atrakcij, živalskih vrtov in hostlov, kjer je shema podatkovne baze predstavljena na sliki 11.1.

Slika 11.1: Shema podatkovne baze problemske domene v okviru DN

V okviru domače naloge dopolnite obstoječo implementacijo spletne prodajalne, kot zahtevajo navodila. Domača naloga zahteva poznavanje večnivojske arhitekture informacijskega sistema, s tehnološkega vidika pa ukazov git, označb HTML, stilov CSS, jezika JavaScript, Node.js tehnologije z izbranimi knjižnicami in XML zapis podatkov. Pri delu natančno sledite navodilom in poskrbite, da bodo vse obstoječe funkcionalnosti delovale tudi po implementaciji vaših nalog!

Opozorilo: Vse oddane domače naloge (1. del) bodo pregledane z vidika medsebojne vsebinske podobnosti in v primeru suma prepisovanja bodo izbrani študenti (poleg nekaj naključno izbranih) pozvani na dodatni zagovor domače naloge. Pri ugotavljanju prvotnega avtorstva oddane rešitve se upošteva prva oddana rešitev študenta (glede na čas uveljavitve na GitHub-u).

Kako naj oddam nalogo?

Spletno povezavo na GitHub repozitorij oddate v obliki https://github.com/ois-2022-2023/DN-{študent}, kjer {študent} predstavlja vaše GitHub uporabniško ime. To spletno povezavo oddate le kot odgovor na nalogo na spletni učilnici.

Vsako nalogo boste izdelali v svoji veji razvoja, spremembe smiselno uveljavljali in vse uveljavitve tudi prenesli v oddaljeni repozitorij v istoležne veje. Na koncu boste vse veje ustrezno združili v vejo main. Vej po združevanju ne brišite!

Bodite pozorni, da natančno upoštevate navodila, saj bodo oddaje avtomatsko preverjene (zadnje uveljavitve v vejah stilizacija, funkcionalnosti in main) in zaradi nepravilne oddaje posledično ne boste dobili točk.

Opozorilo: Število uveljavitev ne vpliva na končno oceno, je pa zahtevano, da v okviru vsake naloge izvedete vsaj 2 uveljavitvi (začetno in končno), kot je to podrobneje predstavljeno v naslednjem odstavku. V okviru posamezne naloge (v vsaki veji razvoja) se bo namreč preverjalo, kaj ste implementirali med prvo in zadnjo uveljavitvijo.

11.1.2 Vzpostavitev repozitorija

Na GitHub-u, v okviru organizacije našega predmeta ois-2022-2023, dostopajte do začetne verzije vašega repozitorija DN-{študent}, kjer {študent} predstavlja vaše GitHub uporabniško ime.

Z uporabo podanih git ukazov v ukazni lupini lokalnega razvojnega okolja vzpostavite lokalni repozitorij, dodajte datoteko DestinationInvoice.sl3 v .gitignore (spremembe v podatkovni bazi ne boste posredovali v GitHub repozitorij) in ustvarite veje razvoja stilizacija in funkcionalnosti, v okviru katerih boste reševali posamezne naloge.

Ukazi, s katerimi pričnete reševanje:

git clone https://github.com/ois-2022-2023/DN-{študent}.git
cd DN-{študent}
npm install

Uporabniki operacijskih sistemov Linux ali Mac OS zaženite sledeč ukaz:

echo -e "\nDestinationInvoice.sl3" &>> .gitignore

Uporabniki operacijskih sistemov Microsoft Windows zaženite sledeč ukaz:

echo DestinationInvoice.sl3 >> .gitignore

Opomba: Če dodajanje datoteke DestinationInvoice.sl3 v ukazni vrstici v okolju Windows ne deluje, jo dodajte ročno.

Ne glede na operacijski sistem zaženite še sledeče ukaze:

git rm -r --cached DestinationInvoice.sl3
git commit -am "Ne upoštevaj sprememb v podatkovni bazi"
git commit --allow-empty -m "Začetek dela na domači nalogi"
git checkout -b stilizacija
git commit --allow-empty -m "Začetek dela na I. nalogi"
git checkout main
git checkout -b funkcionalnosti
git commit --allow-empty -m "Začetek dela na II. nalogi"
git checkout main
git push --all

Opomba: Zelo pomembno je, da vse omenjene ukaze izvedete takoj na začetku reševanja naloge!

11.1.3 Skladnja strani in stilska preobrazba

Trenutna verzija spletne aplikacije je potrebna stilske preobrazbe s pomočjo programskih jezikov HTML, CSS in JavaScript. Sledeče funkcionalnosti podprite v veji razvoja stilizacija, kjer sledite naslednjim navodilom.

Namig: V vejo razvoja stilizacija se prestavite z ukazom git checkout stilizacija.

V tej nalogi nadgradite spletno aplikacijo tako, da implementirate naslednje funkcionalnosti:

  • S pomočjo CSS poskrbite za sledečo stilsko preobrazbo:
    • Na spletnem naslovu /prijava HTML elementu input in vsem HTML elementom select dodajte rdeč rob in sivo senco kot je prikazano na sliki 11.2. V ta namen v stilski datoteki public/slogi/layout.css najprej definirajte CSS razreda rob in senca, ki ju najdete v gradivu iz vaj V3.

    Slika 11.2: Robovi in sence

    • Na spletnem naslovu /prijava dodajte edinemu tekstovnemu vnosnemu polju (HTML element input) spodnji odmik za 14px.
    • Na spletnem naslovu / dodajte HTML elementu z enoličnim identifikatorjem mapa_id zgornji odmik 20px.
    • V CSS datoteki public/slogi/prijava.css dodajte obstoječemu HTML elementu table sredinsko poravnavo tako, da se bo tabela na spletnem mestu /prijava poravnala na sredino stolpca pod naslovom Statistika v tabeli.
    • Tabeli na spletnem mestu /prijava je potrebno pripraviti temeljito stilsko posodobitev (glej sliko 11.3), in sicer:
      • Celicama s tekstom Destinacije in Lastnosti popravite barvo pisave v temno sivo (darkgrey) in poravnajte ležeče.
      • HTML elemente td nizov Turistične atrakcije, Živalski vrtovi, Hostli, Vsi, Izven Ljubljane, Spletna stran in Wikidata zamenjajte z naslovnimi elementi tabele.
      • V datoteki public/slogi/prijava.css popravite obstoječi razred diagonala tako, da bo prva polovica celice v levem zgornjem kotu po diagonali imela barvo ozadja #E4FDEA, druga polovica celice v desnem spodnjem kotu pa barvo ozadja #CCF3D5.
      • Celica tabele s nizom Lastnosti naj se razprostira vertikalno čez vse vrstice tabele in naj bo poravnana po sredini vertikalno.
      • Celica tabele s nizom Destinacije naj se razprostira horizontalno čez zadnje tri stolpce in naj bo poravnana horizontalno po sredini.
      • Prvi vrstici dodajte barvo ozadja #E4FDEA, drugi vrstici barvo ozadja #CCF3D5 ter spodnji črtkan rob zelene barve (green) velikosti 3px in od vrstic 3 do 6 prvih stolpcev dodajte barvo ozadja #CCF3D5 z desnim črtkanim robom zelene barve (green) velikosti 3px. Preostalim celicam desno in spodaj od črtkanega roba zapolnite s polnimi robovi velikosti 1px in temno sive barve (darkgrey).
      • Vsem HTML elementom td dodajte odmike v vse smeri velikosti 3px ter vsem HTML elementom th odmike v vse smeri velikosti 6px.

11.2.2 Opis osnovne ideje predloga

Vaša funkcionalnost je namenjena prikazu podatkov na grafu, uporablja zunanji vir in uporablja še vsaj 3 poljubne HTML komponente. Teaching API lahko uporabite, za branje obstoječih podatkov, dodajanje in posodabljanje. Ideja, način delovanja in uporaba HTML komponent je prepuščena vam.

Predlog rešitve naj bo vaš, kjer si morate zamisliti scenarij uporabe, uporabnika (npr. neprijavljen uporabnik oz. gost ali prijavljen uporabnik kot stranka iz 1. dela domače naloge) in namen uporabe (npr. obveščanje, informativno, zabava, izobraževanje, igra ipd.).

Funkcionalnost naj, razen morebitne prijavljene stranke iz obstoječe aplikacije, ne uporablja lastne podatkovne baze, ampak naj zgolj dostopa do podatkov (bere in ažurira) iz naslednjih podatkovnih virov:

  • podatki o uporabniku (in morebitnih ostalih podatkov) iz platforme Teaching API,
    • za vsakega študenta je pripravljena lastna podatkovna baza z enoličnim identifikatorjem, enakim GitHub uporabniškemu imenu,
  • podatki na Sensei verigi blokov (podrobnosti v 11.2.5),
    • račun si kreirate sami in zahtevate začetna sredstva, kot je opisano v 11.2.1,
  • 1 zunanji podatkovni vir, ki je neposredno ali posredno povezan z vašo izbrano domeno,
    • npr. Top 15 free APIs, Big List of Free and Open Public APIs, 9 Free Public APIs That Offer Up Some Cool Open Data, NASA APIs, OPSI - Odprti podatki Slovenije idr.

Pri integraciji z zunanjim virom se predvideva, da boste komunicirali preko API-ja, pogojno pa lahko v svoj repozitorij prenesete tudi podatke v datoteki (npr. seznam v obliki datoteke, ki ga vaša funkcionalnost uporablja, kjer je dodatno opisan postopek predhodnega procesiranja podatkov oz. še bolje, datoteko prenesete v Teaching API podatkovno bazo, v okviru vašega dostopa).

Kaj naj oddam?
